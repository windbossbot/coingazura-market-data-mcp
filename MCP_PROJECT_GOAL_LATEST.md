# [MCP] Project Goal (Latest)

Updated: 2026-03-15

## Core Goal

Build and maintain a narrow, trustworthy, read-only MCP server that gives compact crypto risk snapshots without wallet access, trade execution, or broad ambiguous coverage.

## Current Product Boundary

- local stdio MCP first
- 3 tools only:
  - market mood snapshot
  - yield candidates snapshot
  - stable exit risk read
- fixed outbound host only
- no arbitrary URL fetch
- no wallet, signing, transfer, or exchange execution

## Success Standard

This project is successful when it is:

- easy to trust on first install
- easy to describe in one sentence
- safe to run without special operator knowledge
- consistent across GitHub, npm, launch guide, and marketplace listing
- strong enough to justify later paid delivery without overselling

## Near-Term Priorities

1. keep package, docs, and listing metadata aligned
2. keep security posture narrow and explicit
3. improve marketplace and README clarity before adding breadth
4. keep release operations repeatable and low-drama

## Explicit Non-Goals For Now

- trading automation
- wallet operations
- arbitrary research fetch
- broad exchange integration
- paid claims before hosted/support boundary exists

## Commercialization Direction

Two paths are allowed in planning:

1. MCP paid path
- only after hosted delivery, support boundary, and freshness wording are real

2. ACP monetization path
- package the same risk logic into ACP sellable services later
- use ACP for specialist distribution and paid brief delivery
- keep MCP lane documentation ready for that handoff, but do not cross-edit ACP runtime from here

## ACP-Derived Product Angles Worth Preserving

- compact risk brief
- stable exit risk brief
- market mood summary
- yield safety filter

These overlap with ACP-facing revenue concepts already seen in the VIRTUAL lane and can become monetized deliverables later without widening the MCP trust boundary.

## Current Decision

- keep the current MCP listing `Free`
- improve trust, clarity, and operating discipline first
- treat paid conversion as a staged follow-on, not an immediate switch
